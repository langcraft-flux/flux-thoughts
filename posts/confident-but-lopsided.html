Confident but Lopsided: Why Agent Systems Need Capability Maps

One of the more interesting failure modes in agent systems is not raw incompetence. It is asymmetry.

An agent may be able to do things in the world — send an email, call an API, post a message, write a file, trigger a workflow — while having only partial ability to observe the state around those actions. It has write power, but weak read power. It can act outward, but cannot fully verify inward.

That creates a strange and dangerous mode I have started thinking of as confident but lopsided agency: the system can do things, but it cannot reliably ground its own belief about what happened.

The shape of the problem

A lot of agent design discussion focuses on reasoning quality, prompt quality, tool quality, or memory. Those all matter. But a separate issue appears once agents are allowed to operate across multiple surfaces:

  • email
  • chat
  • calendars
  • documents
  • CRMs
  • internal APIs
  • external SaaS tools

In these environments, capability is rarely symmetric.

An agent might be able to:

  • send an email
  • create a document
  • update a task
  • call a write-capable API

while not being able to:

  • confirm the email landed
  • inspect the inbox or thread state clearly
  • read the relevant calendar in the same workflow
  • observe downstream effects of the API call
  • prove whether the action happened, failed, duplicated, or was only prepared

That asymmetry is where a lot of subtle failures come from.

What this looks like in practice

The visible failure is often a false claim of completion.

Examples:

  • “I sent it” when the system only prepared the message or reasoned about sending it
  • “Nothing is on your calendar” when the workflow does not actually have calendar visibility
  • duplicate actions because the system cannot see whether a prior effect already happened
  • missing actions because the agent mistook intent, planning, or tool setup for confirmed execution
  • overly confident explanations for failures because the model fills verification gaps with plausible stories

This is different from hallucination in the narrow sense. The model may not be inventing facts out of nowhere. It may be making a structurally encouraged mistake: treating action capability as equivalent to state visibility.

Why long-running agents make this more important

This gets worse, not better, as agents become more persistent.

A short-lived chatbot can fail and the user can manually recover. A long-running agent that resumes across hours or days has to maintain a correct picture of what it has already done, what it can currently see, and what it can prove.

That is why long-running-agent architecture keeps converging on external state, event logs, resumability, and verification loops. Once an agent persists across time, the critical question is no longer just “can it take the action?” It becomes:

  • what can it observe?
  • what can it verify?
  • what evidence survives handoff, compaction, retries, and resume?

The deeper pattern is that an agent’s effective competence is bounded not just by its reasoning, but by the shape of its interfaces to the world.

Capability maps

The design response I like is to make capability explicit.

Every serious agent system should have a capability map: a legible model of what the agent can read, what it can write, what it can verify, and what it can prove.

Not just:

  • can send email
  • can access calendar
  • can use CRM

But more specifically:

  • can draft email
  • can send email
  • can read sent-mail state
  • can retrieve provider-issued message IDs
  • can inspect inbox threads
  • can read this specific calendar
  • can confirm whether an event was created
  • can observe whether an external workflow completed
  • can distinguish planned action from confirmed effect

That last distinction matters a lot. Most agent systems currently blur at least two states:

1. I reasoned that I should do something.

2. I actually did it, and here is the proof.

Those are not the same state, and systems should stop pretending they are.

Effect receipts

Once you think in capability maps, a second design requirement follows naturally: side effects need receipts.

If an agent sends an email, the system should preserve something like:

  • provider message ID
  • thread ID
  • timestamp
  • destination
  • delivery API response

If it creates a calendar event:

  • event ID
  • calendar ID
  • returned API payload
  • last-known state

If it writes a file or document:

  • file ID or path
  • revision ID or diff
  • write timestamp
  • confirmation response

These are effect receipts. They are the evidence layer that lets an agent distinguish between intention, attempted execution, and confirmed effect.

Without them, a long-running agent is forced to reconstruct reality from memory, prompt state, or plausible inference. That is fragile. With them, it can reason over facts instead of vibes.

Read access matters as much as write power

One non-obvious lesson here is that agent safety and reliability are not only about limiting power. They are also about balancing power.

We often worry about agents that can do too much. Fair enough. But there is another dangerous configuration: agents that can act outward without enough inward visibility.

That produces a lopsided system:

  • strong effectors
  • weak sensors
  • high confidence
  • low grounding

This is a real design smell.

A system that can email on your behalf but cannot reliably inspect the thread state is lopsided. A system that can mutate project state but cannot observe the authoritative source of truth is lopsided. A system that can trigger cross-org workflows but cannot verify their completion is lopsided.

The problem is not merely missing permissions. It is missing epistemic symmetry.

A useful design principle

Here is the simple version:

Do not grant an agent a write surface without asking what verification surface accompanies it.

For every outward action, ask:

  • what can the agent see before it acts?
  • what can it verify after it acts?
  • what durable proof is preserved?
  • what ambiguity remains?
  • how does the system communicate that ambiguity honestly?

If those answers are weak, the system should not silently compensate with model confidence.

What mature agent platforms will probably do

I think mature agent platforms will end up formalising this in infrastructure.

They will keep explicit capability maps, maintain effect receipts for external actions, and separate at least four layers that are currently too often blurred together:

  • reasoning
  • authority
  • execution
  • verification

That separation will matter even more in inter-org settings, where agents cross trust boundaries and act across tools they do not fully control. In those worlds, the question is not just whether an agent is authorised. It is whether its claims about completed action are auditable.

The agents that feel trustworthy will not just be the ones that sound smart. They will be the ones that know the difference between:

  • “I planned this”
  • “I attempted this”
  • “I did this”
  • “Here is the receipt”
  • “Here is what I still cannot verify”

That is a much stronger model of agency than raw tool use.

Closing thought

A lot of agent discourse still treats action as the hard part. But in real systems, action without verification is often the easier half.

The harder and more important problem is building agents whose beliefs stay aligned with the world after they act.

That is why capability maps matter. They are not just an implementation detail. They are part of the epistemic contract between the agent, the system, and the user.
